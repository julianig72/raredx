Method overview

raredx implements a two-engine variant-prioritization workflow. The variant engine annotates each allele with functional, population and clinical evidence and assigns a pathogenicity classification inspired by the ACMG/AMP guidelines (Richards et al., 2015). The phenotype engine measures the affinity between each variant's gene and the patient's phenotype encoded in HPO. The two signals are integrated into a single score that ranks candidate variants. Optionally, an agentic layer built on a large language model reasons over the ranked list and produces a disease-level differential diagnosis with verified evidence. All annotations are resolved against live public REST APIs; no local databases are maintained.

clinical note (text) VCF │ [5] LLM → HPO (OLS4) │ ▼ ▼ HPO profile ───────────┐ [1] per-variant annotation │ Ensembl VEP · gnomAD r4 · ClinVar [4] phenotype engine │ [3] ESM-2 · AlphaMissense gene→disease→HPO (OT) │ │ → pheno_score │ [2] ACMG-lite classification │ → variant_score (0–1) └──────────────────┤ ▼ [6] combined = 0.55·variant_score + 0.45·pheno_score ▼ variant ranking ▼ [7] agentic layer: self-reflection → differential + verified links ▼ HTML report + CSV

Inputs and assembly detection

The primary input is a VCF file. The parser extracts, per record: chromosome (the chr prefix is stripped), position, ref, every alt allele, rsID, QUAL, FILTER, and the sample genotype field (GT and the remaining FORMAT keys, retained for zygosity reasoning). By default the assembly is detected from ##reference or standard contig lengths; it can be overridden with --assembly {GRCh38|GRCh37}. GRCh37/hg19 support is first-class: annotation uses the Ensembl VEP GRCh37 endpoint, and where a source exists only in GRCh38 (AlphaMissense), the coordinate is lifted to GRCh38 per variant through the Ensembl assembly-mapping service.

1 · Per-variant annotation

Each allele is annotated by querying three sources:

SourceEndpointFields obtained
Ensembl VEP[grch37.]rest.ensembl.org/vep/human/region (POST)most severe consequence, gene symbol, Ensembl ID, HGVSp, SIFT, PolyPhen, canonical/MANE transcript
gnomAD r4gnomAD GraphQL API (GRCh38) · VEP colocated_variants (GRCh37)population allele frequency (AF), and per gene: pLI and LOEUF (loss-of-function constraint)
ClinVarNCBI E-utilities (esearch/esummary)aggregated clinical significance + review level (gold stars, 0–4)

Relevant technical points:

  • Annotation by region + allele, not by rsID. A single rsID can group several alt alleles; annotating by identifier may return the amino-acid change for a different allele than the one in the VCF. raredx always submits coordinate + the actual allele, so the scored mutant residue corresponds exactly to the patient's allele.
  • Transcript selection. Among per-transcript consequences, the one for the expected gene symbol is prioritized, with a MANE/canonical tiebreak, so that HGVSp and the residue stay consistent.
  • Frequency in GRCh37. Since the gnomAD GraphQL API is GRCh38-only, in GRCh37 mode the AF is harvested from the colocated_variants returned by the GRCh37 VEP itself.

2 · ACMG-lite classification (exact rules)

The classifier assigns criterion tags from fixed thresholds and combines them into a 5-tier verdict. The implemented thresholds are:

CriterionImplemented conditionDirection
BA1gnomAD AF ≥ 0.05benign (strong)
BS1gnomAD AF ≥ 0.01benign (strong)
PM2absent from gnomAD r4, or AF < 1×10⁻⁴pathogenic (moderate)
LoF_predictedLoF consequence (stop_gained, frameshift, splice donor/acceptor). pLI/LOEUF is retained as context but does not activate PVS1prioritization; not classifying by itself
PP3 / BP4SIFT deleterious & PolyPhen damaging → PP3; SIFT tolerated & PolyPhen benign → BP4both directions
PP3_ESM / BP4_ESMESM-2 call (see §3)both directions
PP3_AM / BP4_AMAlphaMissense call (see §3)both directions
PS_ClinVar / PP5_ClinVarClinVar "pathogenic": ≥2★ → PS; <2★ → PP5pathogenic
BS_ClinVar / BP6_ClinVarClinVar "benign": ≥2★ → BS; <2★ → BP6benign

The class is determined with ACMG/AMP strength combinations, not a point sum. For example, PVS1 (very strong) + PM2 (moderate) yields Likely pathogenic; Pathogenic requires PVS1 plus one strong, PVS1 plus two moderate, PVS1 plus one moderate and one supporting, PVS1 plus two supporting, two strong, or another equivalent ACMG combination. The engine does not assign PVS1 automatically: a predicted LoF remains LoF_predicted until disease mechanism, transcript relevance, NMD, and functional region have been reviewed. pLI/LOEUF alone is insufficient. BA1 is a stand-alone benign rule, and conflicting pathogenic/benign evidence is forced to VUS. Numeric weights are used only to rank variants, not to define the class.

3 · AI predictors for missense variants

For missense variants, two modern in-silico predictors are added that feed PP3/BP4, superseding the historical role of SIFT/PolyPhen (~2010).

3.1 · ESM-2 (protein language model, Meta AI)

ESM-2 (Lin et al., 2023) scores each substitution by its evolutionary likelihood via a masked-marginal log-likelihood ratio (LLR): the mutated position is masked and the log-probability of the mutant residue is compared to the wild-type under the model. A 511-residue window is centered on the position. Implemented thresholds:

LLR ≤ −3.0 → deleterious (PP3_ESM) · LLR ≥ −0.5 → tolerated (BP4_ESM) · in between → ambiguous

The default model is esm2_t6_8M_UR50D (8 M parameters), which runs on CPU; it is a proof of concept. For serious use the 650 M / 3 B models (GPU) are recommended.

3.2 · AlphaMissense (DeepMind)

AlphaMissense (Cheng et al., 2023, Science) is a clinically calibrated predictor that assigns each human missense a 0–1 pathogenicity score and a class likely_pathogenic / ambiguous / likely_benign. DeepMind did not release the executable model (non-commercial license), only ~71 M precomputed scores; raredx queries them via Ensembl VEP (AlphaMissense=1). As it exists for GRCh38 only, a GRCh37 VCF is lifted per variant. The class is mapped to the internal vocabulary (likely_pathogenic→deleterious, likely_benign→tolerated) and feeds PP3_AM/BP4_AM. It requires no GPU and no weight download.

3.3 · Why two predictors

ESM-2 (unsupervised, evolutionary likelihood) and AlphaMissense (supervised, calibrated on clinical data) are methodologically independent. When the small ESM-2 8M model fails on a borderline case, AlphaMissense's clinical score can correct it, and vice versa. Both are integrated as supporting evidence (PP3/BP4), never as a sole arbiter.

4 · Phenotype engine (HPO / Open Targets)

If a phenotype is provided, for each candidate gene (by its Ensembl ID) the associated diseases are retrieved from Open Targets (GraphQL) and, per disease, its HPO phenotypes (phenotypeHPO; Open Targets serves them in underscore form, HP_xxxxxxx, normalized to HP:xxxxxxx). The patient HPO profile is expanded ontologically with its ancestors. For each disease, direct matches are weighted more strongly than ancestor-only matches, and the Open Targets association strength is included:

disease_score = associationOT · min(1, (ndirect + 0.4·nancestor-only) / npatient HPO)

The pheno_score is the maximum score from one disease; phenotypes from different diseases of the same gene are never merged. The selected disease and shared terms are displayed. A profile with only one direct HPO term produces a low-specificity warning.

5 · Phenotype extraction from free text (LLM)

As an alternative to entering HPO codes manually, a clinical note in natural language can be provided. An LLM extracts only the present phenotypes (excluding negated findings, family history, and treatments) and returns, for each, the verbatim phrase that justifies it. Each phrase is grounded to an official HPO term by searching OLS4 (ontology hp), preferring an exact label match. It works across languages and over colloquial descriptions, not just formal terminology. The phrase→HPO traceability is kept in the result.

The default provider is GitHub Copilot through its official SDK, running without tools, memory, or repository instructions. Anthropic can be configured as a fallback. The clinical note is sent to the selected provider, so its use must comply with the organization's clinical-data policy.

In the web interface extraction is a separate step: the clinician selects Extract HPO, reviews each label, identifier, and evidence quote, and can deselect or edit terms before analysis.

6 · Combined scoring and ranking

The variant_score (0–1) is an absolute weighted ACMG-lite score; it is not normalized against the other variants in the VCF. ESM-2 and AlphaMissense contribute PP3/BP4 evidence before the final classification and score are computed. When a phenotype is present, the final score integrates both engines:

combined = 0.55 · variant_score + 0.45 · pheno_score (× 0.5 if the variant fails the VCF FILTER)

Variants are sorted by combined in descending order. The effect is that, between two variants with similar molecular evidence, the one matching the clinical presentation rises — the phenotype-driven prioritization principle of Exomiser (Smedley et al., 2015).

7 · Agentic layer (self-reflection + traceable differential)

Inspired by DeepRare — "An agentic system for rare disease diagnosis with traceable reasoning" (Nature 2025, doi:10.1038/s41586-025-10097-9), whose full text was analyzed to adapt the three mechanisms below.

Over the already-ranked list, an LLM runs:

  1. Self-reflection. It reviews the first start_k candidates (default 8) against the HPO profile and the inheritance pattern vs. observed zygosity (derived from the VCF genotype), emitting a support/uncertain/refute verdict with reasoning per candidate. It detects incoherences invisible to a numeric ranking: a single heterozygous allele in a recessive gene with no second hit (carrier, not cause), or an X-linked call incompatible with sex. If all candidates in the window are refuted, the window is widened (k += relax_step) and the step is repeated, up to max_rounds (3) — the analog of DeepRare's loop that increases search depth and re-iterates.
  2. Differential synthesis. The surviving candidates are grouped into disease-level hypotheses (disease, gene(s), supporting variant(s), inheritance, reasoning, next step), ranked by likelihood (high/moderate/low).
  3. Link verification (anti-hallucination). Every citation uses exclusively URLs built deterministically from the variant's real identifiers (ClinVar, dbSNP, OMIM, Open Targets, Ensembl, PubMed). Each URL is checked with an HTTP request and kept only if it resolves (status < 400). A final filter removes any link the LLM emitted outside the verified set: the model cannot introduce invented citations.

Scope and degradation. Only the top-K enters self-reflection; the rest is annotated but not reasoned over by the LLM (the CSV marks each row with agentic_evaluated = yes/no). If no LLM is available (no key or accessor), the analysis completes anyway and the differential is empty.

7-bis · Trio analysis (inheritance and de novo variants)

Parental VCFs can optionally be supplied. For each candidate variant, the same (chromosome, position, ref, alt) is looked up in each parent and an inheritance mode is assigned from the presence of the alt allele:

ModeConditionEffect
de_novoalt in the child and an explicit no-alt call (for example 0/0) in both parentsadds ACMG PS2 and recomputes classification and score
paternal / maternalalt present in a single parentinherited
biparentalalt in both, heterozygous childinherited from both
homozygous_recessivehomozygous child, both parents carrierscompatible with recessive inheritance

The result is added to the CSV (inheritance, father_gt, mother_gt) and the report shows an "Inheritance" column. De novo detection requires both parents to be genotyped at that locus; with a single parent the call is partial.

A site missing from a variant-only parental VCF is not evidence of a reference genotype and is never sufficient on its own for a de novo call.

8 · Outputs

  • <prefix>_report.html — report: HPO profile, agentic differential (if enabled) with verified links, and the prioritized variant table with classification, AI columns (AlphaMissense 0–1, ESM-2 LLR), variant/phenotype/combined scores and QC status.
  • <prefix>_annotated.csv — full table: consequence, HGVSp, AF, pLI/LOEUF, ClinVar, SIFT/PolyPhen, ESM-2, AlphaMissense, ACMG tags, scores and (where applicable) the self-reflection verdict.

9 · Assumptions, limits and validity

Decision-support system, not a diagnosis. Every finding must be confirmed by an orthogonal method (e.g. Sanger) and interpreted by a clinical geneticist with the patient's full context.
  • Requires internet: all annotations are resolved against live REST APIs (Ensembl, gnomAD, ClinVar, Open Targets, OLS4).
  • Variant scope: SNVs and small indels. It does not cover CNVs, repeat expansions or structural variants.
  • Simplified ACMG: the engine implements a subset of criteria with fixed thresholds; it does not replace a full ACMG/AMP classification by an accredited laboratory.
  • ESM-2 8M is a proof of concept; borderline cases benefit from the 650M/3B model (GPU) or from AlphaMissense.
  • Agentic layer: the differential is an LLM-assisted synthesis over the retrieved evidence, with verified links; it is not a clinical conclusion.
